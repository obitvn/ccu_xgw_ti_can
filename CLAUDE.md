# CCU Multicore Project - Development Guidelines

## Project Overview

**Project Name:** CCU TI Multicore System
**Platform:** TI AM263P4 (Dual R5F Cores)
**SDK:** `D:\VinDynamics\Project\CCU_TI_RS485_CAN\2.Firmware\draft\mcu_plus_sdk_am263px_11_01_00_19`

**Purpose:** Ethernet-to-CAN gateway for VD1 robot control. Receives motor commands via UDP (xGW protocol) and transmits to 23 motors across 8 CAN buses at 1000Hz. Handles IMU sensor data transmission.

**Architecture:** Dual-core asymmetric multiprocessing:
- **Core0 (FreeRTOS)**: Ethernet Gateway - CPSW Ethernet, lwIP stack, xGW UDP protocol
- **Core1 (NoRTOS/Bare-metal)**: Real-time Control - 8x MCAN buses, 1000Hz motor control loop, IMU UART ISR

## Architecture Constraints

### Core Assignment
```

- Core0 (R5FSS0-0): FreeRTOS + Ethernet (ccu_ti_multi_core_freertos)
- Core1 (R5FSS1-1): NoRTOS + CAN (ccu_ti_multi_core_realtime)

```



### IPC Boundary
- **Method:** Shared memory + Lock-free ring buffer
- **Notification:** IpcNotify (interrupt-based)
- **Client ID:** GATEWAY_IPC_CLIENT_ID = 4 (BOTH cores use same ID)
- **KHÔNG share global variables trực tiếp** - chỉ qua shared memory interface ringbuffer

### Core1 Constraints (NoRTOS)
- **NO malloc** - All memory static or stack
- **NO printf** - Use DebugP_log (UART)
- **NO OS calls** - No FreeRTOS APIs
- **Debug visibility:** Shared memory counters + DebugP_log

### Data Flow
```
PC (UDP) --[xGW protocol]--> Core0 (FreeRTOS)
                                       | ^
                                       v |
                              Shared Memory (Ring Buffer)
                                       | ^
                                       v |
Core1 (NoRTOS) --[8x CAN]--> 23 Motors (Robstride)
                  ^
                  |
            IMU UART ISR
```

## xGW UDP Protocol

**Specification:** `xGW-Linux UDP Protocol Specification.html`

**Ports:**
- UDP RX (PC → CCU): Port 61904 - Motor commands
- UDP TX (CCU → PC): Port 53489 - Motor states

**Message Types:**
- `XGW_MSG_TYPE_MOTOR_CMD` (0x01) - Single motor command
- `XGW_MSG_TYPE_MOTOR_SET` (0x02) - Multiple motor commands
- `XGW_MSG_TYPE_MOTOR_STATE` (0x03) - Motor state response

**Motor Units (IPC format → xGW format):**
- Position: 0.01 rad → rad (divide by 100)
- Velocity: 0.01 rad/s → rad/s (divide by 100)
- Torque: 0.01 Nm → Nm (divide by 100)
- Temperature: 0.1°C → °C (divide by 10)

**CRC:** CRC32 verification on all packets

## Debug Strategy

### Common Issues

| Symptom | Possible Cause | Check |
|---------|---------------|-------|
| No UDP RX | Ethernet not up, PHY not linked | Check `xgw_udp_is_initialized()`, PHY status |
| CAN not sending | Ring buffer empty, Core0 not sending | Check `g_commands_ready`, IPC notification |
| Hardfault | Memory access violation, stack overflow | Check fault handler PC, stack usage |
| Timing miss | 1000Hz loop blocked | Check timer ISR count, g_cycle_count |
| IPC timeout | Shared memory not initialized | Check `gateway_core1_wait_for_ready()` |
